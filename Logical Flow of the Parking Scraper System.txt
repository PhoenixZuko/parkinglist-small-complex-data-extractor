**Logical Flow of the Parking Scraper System**

---

### 1. **Start: `main.py`**

* The main script is executed to initiate the process.

---

### 2. **Read Configuration from `included_airports.yaml`**

* This file contains instructions for enabling/disabling specific airports.
* Based on the configuration, the list of active airports is generated and saved to:

  * `core/airports.txt`

---

### 3. **Iterate Through Airports (`airports.txt`)**

* For each airport URL:

  * The program generates combinations of dates using the logic:

    ```python
    durations = range(1, 15)  # 14 durations
    for n in range(1, 6):     # 5 start dates from today
    ```
  * This results in **70 combinations per airport**.

---

### 4. **Process Each Date Combination**

* Accesses the airport page using undetected Chrome.
* Saves each resulting HTML page in:

  * `saved_pages/`
* Saves the processing status in:

  * `progress.log`
  * Format: `airport_url|from_date|to_date`

---

### 5. **After Completing All Combinations for an Airport**

* The airport URL is removed from `airports.txt`
* The script continues with the next airport.

---

### 6. **If the Program is Interrupted**

* Upon restart:

  * Checks which combinations were already processed (via `progress.log`)
  * Resumes from the exact point it was interrupted

---

### 7. **When All Airports and Dates Are Completed**

* The `finalizer.py` script is triggered:

  * Verifies that `airports.txt` is empty and `progress.log` is not.
  * If so:

    * Clears `progress.log`
    * Executes `extract_parking_data.py`

      * Parses all HTML files from `saved_pages/`
      * Saves results in `json_out/parking_data.json`
    * Moves all processed `.html` files to:

      * `old_saved_pages/YYYY-MM-DD_HH-MM-SS/`

---

### 8. **Result**

* The system can restart fresh if completed.
* If interrupted, it resumes without losing progress.
* Organized data flow and archiving ensure reliability and clarity.



Logical Flow – Parking Scraper System
Overview

This system is designed as a resumable, multi-stage extraction pipeline, optimized for long-running execution and fault tolerance.
It ensures that progress is never lost and that data processing remains consistent and repeatable.

1. Entry Point – main.py
The execution starts from main.py.
This script initializes the scraping workflow and orchestrates the entire process.
2. Configuration Layer – included_airports.yaml
Defines which airports are enabled or disabled.
Based on this configuration, the system generates the active airport list:
core/airports.txt
3. Airport Iteration
The system processes each airport sequentially from airports.txt.
For every airport, it generates date combinations:
durations = range(1, 15)  # 14 durations
for n in range(1, 6):     # 5 start dates
Result:
70 combinations per airport
4. Extraction Phase

For each combination:

Opens the page using undetected Chrome automation
Saves raw HTML to:
saved_pages/
Logs progress in:
progress.log

Format:

airport_url | from_date | to_date
5. Completion per Airport
Once all combinations are processed:
The airport is removed from airports.txt
The system proceeds to the next airport
6. Resume Mechanism (Critical Feature)

If execution is interrupted:

The system reads progress.log
Skips already processed combinations
Continues exactly from the last unprocessed state

This ensures:

no duplicate processing
no data loss
efficient recovery after failure
7. Finalization Phase – finalizer.py

Triggered when all airports are completed:

Validates:
airports.txt is empty
progress.log contains completed entries

Then:

Clears progress.log
Executes:
extract_parking_data.py

Which:

Parses all HTML files
Extracts structured data
Saves output:
json_out/parking_data.json
Archives processed files:
old_saved_pages/YYYY-MM-DD_HH-MM-SS/
8. Output & System State
Final structured dataset available in JSON format
HTML files safely archived
System reset for next execution cycle
Key Design Principles
Resumable execution → no loss of progress
Separation of stages → extraction, parsing, finalization
Deterministic processing → same input → same output
Fault tolerance → safe recovery from interruptions
Clean data lifecycle → raw → processed → archived
Summary

The system behaves as a robust data pipeline, not a simple scraper:

generates controlled extraction scenarios
tracks progress persistently
supports interruption and recovery
processes and structures data in stages
maintains clean and auditable outputs
